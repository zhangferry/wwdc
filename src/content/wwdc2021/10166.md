## 一句话判断

DocC 不只是"Xcode 自带的 Jazzy 替代品"——它是 Apple 把文档编写从"附属品"提升为"一等公民"的尝试，如果你维护一个 Swift 库，没有理由不用它。

## 这场 Session 讲了什么

Session 介绍了 Xcode 13 中全新集成的 DocC（Documentation Compiler）系统。DocC 是 Apple 开发的文档编译器，可以将代码中的 Markdown 格式注释编译成精美的网页文档，支持文章（Articles）、教程（Tutorials）和 API 参考三种内容类型。

过去 Swift 项目的文档方案主要依赖 Jazzy 等第三方工具，输出样式和 Apple 官方文档不一致。DocC 原生集成在 Xcode 中，使用和 Apple 开发者文档完全相同的渲染引擎。你在代码中写的 `///` 注释会被 DocC 自动提取为 API 参考文档。额外的 Article 和 Tutorial 文件用 Markdown 编写，放在项目的 `Documentation` 目录下。编译后的文档可以直接在 Xcode 的 Documentation Window 中查看，也可以导出为静态网站部署到任何 Web 服务器。

## 值得深挖的点

**DocC 的 Markdown 方言（Documentation Markup）**

DocC 使用一种扩展的 Markdown 语法。除了标准的标题、列表、代码块之外，它支持几种特殊结构：`@Code` 引用代码片段、`@Image` 嵌入图片、`@Row` / `@Column` 布局排版、`@TabNavigator` 创建标签页切换。最有用的是 `@Links` 和 `@Reference`——可以创建类型之间的交叉引用。当你的文档提到另一个类型时，DocC 自动生成可点击的链接，不需要手写 URL。

**Tutorial 的分步引导设计**

DocC 的 Tutorial 功能可以创建交互式的分步教学文档。每个步骤包含一段说明文字、一段代码示例和一个 Checkpoint（检查点）。用户可以逐步跟进代码修改，Xcode 会在 Documentation Window 中高亮显示当前步骤的代码变化。这个功能非常适合编写框架的入门教程——比单独维护一个 GitHub Wiki 好太多。

## 最佳实践

- 从第一天就开始写文档注释——不要等项目完成了再补。`///` 注释写在代码旁边，更新代码时顺手更新文档，成本几乎为零。
- 为公开 API 提供至少一个使用示例代码块。DocC 会把代码块中的类型名自动转换为可点击的链接，用户可以在文档中直接跳转。
- Tutorial 文件用 `@Steps` 和 `@Code` 组织，保持每个步骤只讲一个概念。长篇大论的 Tutorial 比没有文档更糟糕。

## 还有什么值得关注

- DocC 生成的文档可以在 Xcode 的 Documentation Window 中直接搜索，和 Apple 官方文档混合显示。
- Apple 已经开源了 DocC 的编译器和渲染器，社区可以贡献主题和插件。
- `xcodebuild docbuild` 命令支持在 CI 环境中自动编译文档，配合 GitHub Actions 自动部署。
