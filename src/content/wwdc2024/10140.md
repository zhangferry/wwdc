## 一句话判断

这场 Session 不是教你怎么写文案，而是教你怎么用一套系统化的方法——"把 app 想象成一个人"——来定义和执行 app 的写作风格，最终让用户感受到一致且有辨识度的品牌性格。

## 这场 Session 讲了什么

Apple 的人机界面设计团队讲了一个经常被开发者忽视的话题：UX writing 如何塑造 app 的个性。核心框架分三层：personality（个性）是 app 整体的气质，voice（声音）是用文字表达品牌价值观的固定风格，tone（语调）是声音在不同情境下的灵活变化。

Session 用 Apple 自身作为案例。Apple 的 voice 有四个关键词：clarity（清晰）、simplicity（简洁）、friendliness（友好）、helpfulness（有用）。这四个品质贯穿所有 Apple 的 UI 文案，但不同场景下各自的权重不同。庆祝用户完成第 100 次 Fitness+ 训练时，friendliness 拉满——"Don't stop now!"加一个感叹号，像朋友在为你欢呼。通知用户心率异常时，clarity 占主导——"Your heart rate rose above 120 BPM while you seemed to be inactive for 10 minutes"，所有数字和信息直接给到位，不煽情。

讲者提供了两个实操练习：第一个是"把你的 app 想象成一个人"，用便签写下形容词，分组找到关键属性，定义 voice；第二个是用定义好的 voice 写一个欢迎页面。整个过程强调的不是文学才华，而是系统化的方法论。

## 值得深挖的点

### Voice vs Tone：恒定与变化的设计张力

Voice 是恒定的，tone 是变化的。这个区分对开发者来说很实用。很多 app 的文案问题不是"写得不好"，而是没有 voice——不同页面、不同功能的文案风格不统一，用户感受不到这是一个"人"在和他们说话。

定义 voice 的方法是便签分组法。把你的 app 想象成一个人，写下所有能想到的形容词。比如一个投资理财 app 可能写出：smart、serious、motivating、polite、sharp、formal。然后分组：smart/sharp/serious 归为一类，motivating/encouraging 归为另一类。最终提炼出两个核心属性：Smart 和 Motivating——这就是你的 app voice。

而 tone 的调整取决于场景。同样是这个投资 app，用户刚刚完成了一笔定投，tone 可以偏 Motivating："Great job staying consistent!"；用户账户余额不足导致定投失败，tone 就应该偏 Smart/Helpfulness："Your recurring investment didn't go through. Here's what you can do."同样的 voice，不同的 tone。

这个框架对小团队特别有价值。大公司有专门的 UX writer 团队来维护写作风格指南，小团队往往没有这个资源。但便签分组法花一个下午就能完成，产出的 voice 定义可以作为团队共识，减少每次写文案时的犹豫和返工。

### 同一功能，不同 voice 的差异化

Session 展示了一个很直观的对比。同样是"帮助用户存钱"的 app，面向职场新人的投资 app voice 是 Smart + Motivating，面向儿童的存钱 app voice 是 Kind + Fun + Encouraging。功能一样，但欢迎页面完全不同：

投资 app："Start building your future today. Set up your first investment in minutes."——简洁、有方向感。
儿童存钱 app："Let's start saving together! What would you like to save for?"——温暖、互动性、有引导。

这说明 voice 定义不是装饰品，它直接影响 UI 文案的每一个字。如果你发现自己 app 的文案"不知道该用什么语气"，大概率是因为没有做过 voice 定义这个步骤。

## 代码片段

这场 Session 不涉及代码，但以下是一些在 SwiftUI 中落地 UX writing 原则的实用模式。

### 为不同场景配置 tone

```swift
// 根据场景类型返回不同语调的文案
enum MessageTone {
    case celebratory
    case informational
    case error
    case warning
}

struct UXToneHelper {
    // 庆祝场景：friendliness 拉高
    static func congratulation(for achievement: String) -> String {
        return "太棒了！你刚完成了\(achievement)。继续保持！"
    }

    // 信息性场景：clarity 拉高
    static func infoMessage(for event: String, details: String) -> String {
        return "\(event)。\(details)"
    }

    // 错误场景：helpfulness 拉高，避免指责用户
    static func errorMessage(for action: String, solution: String) -> String {
        return "\(action)没有成功。\(solution)"
    }
}
```

### 本地化字符串中保持 voice 一致性

```swift
// Localizable.strings (Base)
"welcome.title" = "Welcome back";
"welcome.subtitle" = "Your plants need watering today";

// Localizable.strings (zh-Hans)
"welcome.title" = "欢迎回来";
"welcome.subtitle" = "你的植物今天需要浇水了";

// 使用时保持一致的 voice
Text(LocalizedStringResource("welcome.title"))
    .font(.title.bold())
Text(LocalizedStringResource("welcome.subtitle"))
    .font(.subheadline)
    .foregroundStyle(.secondary)
```

### 为 onboarding 应用 voice 定义

```swift
struct OnboardingWelcomeView: View {
    // Voice: Kind + Encouraging（面向儿童用户）
    var body: some View {
        VStack(spacing: 20) {
            Image(systemName: "star.fill")
                .font(.system(size: 60))
                .foregroundStyle(.yellow)

            Text("让我们开始一起存钱吧！")
                .font(.title.bold())

            Text("你想为什么东西存钱？")
                .font(.body)
                .foregroundStyle(.secondary)

            Button("开始存钱") {
                // 进入存钱目标设置
            }
            .buttonStyle(.borderedProminent)
        }
        .padding()
    }
}
```

## 最佳实践

**给所有团队的建议：** 花一个下午做一次 voice 定义练习。准备便签，把 app 想象成一个人，写下形容词，分组，提炼出 2-3 个核心属性。把它贴在团队的共享空间里。以后每次写 UI 文案，先问自己"这个语气符合我们定义的 voice 吗"。

**已有项目：** 审视现有 app 的文案，看看不同页面的语气是否一致。特别关注错误提示和空状态——这些是文案最容易出问题的地方。把"操作失败"改成"没有成功 + 具体原因 + 建议操作"，立刻就能提升用户体验。

**新项目：** 在设计阶段就把 voice 定义纳入设计系统。和配色、字体一样，voice 是品牌的一部分。给设计师和开发者一份 voice 指南，让他们在写任何文案时都有参考标准。

## 还有什么值得关注

- Apple 在文案中极少使用感叹号，只有在真正值得庆祝的场景才会出现——如果你的 app 里到处都是感叹号，是时候收敛了
- 便签分组法不仅适用于 voice 定义，也适用于信息架构梳理——强迫你把模糊的想法变成明确的分类
- 同一 app 的不同功能模块可能有不同的 tone，但 voice 应该始终一致——这是"品牌感"的来源
